# 18 — i18n

## Goal

The SPA supports 8 locales (en, es, pt-BR, ko, ja, ru, zh-CN, zh-TW). Switching languages updates every visible string, persists to `localStorage`, and survives a reload.

## Inputs

- `I18n.setLang('<code>')` from the SPA — or the language picker in the topbar.
- Server-side: every LLM route propagates locale via `resolveLocale(req)` (PR-2 / v1.10.1).
- Client-side: `API.call()` auto-attaches `Accept-Language: <code>` and merges `lang: <code>` into JSON bodies.

## Expected outputs

- All keys defined in `public/js/lib/i18n.js` cover all 8 locales.
- `tests/i18n-coverage.test.mjs` fails if a single key is missing for any locale.
- Server prompt builders inject `# Output language\nRespond in <Language> (locale: <code>). Keep code…` when `lang !== 'en'` (PR-2).
- Help bundles served via `/api/help/:lang` resolve in this order: exact match → `FILE_ALIASES[lang]` → language-only base → `en.md` (F-002).

## Negative cases

| Case | Expected |
|---|---|
| Unknown locale code | Falls back to `en` |
| Missing help bundle | Falls back through the resolver chain |

## Test coverage

- `tests/i18n-coverage.test.mjs` — every i18n key × 8 locales.
- `tests/critical-fixes.test.mjs` — `resolveLocale` precedence, `buildLocaleDirective`, prompt builder integration.
- `tests/help-ui.test.mjs` — every locale has a non-empty help bundle.
