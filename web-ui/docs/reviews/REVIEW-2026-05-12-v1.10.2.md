# Code Review — 2026-05-12 (v1.10.2 functional-regression patch)

> Patch release driven by hand-testing the v1.10.1 stand on the user's local machine. Two real bugs found in interactive verification; both shipped in v1.10.0 / v1.10.1, both invisible to the existing automated suite because no test exercised the actual SSE-stream PDF flow end-to-end or sent a `multipart/form-data` upload to `/api/cv/import`.

## Scope (this slice)

- **CV upload regression** — `POST /api/cv/import` accepted multipart uploads and stored the wire envelope as the file content. Bug F-016 was flagged in qa/fixes/ but only partially addressed in v1.10.1.
- **PDF download regression** — `GET /api/stream/pdf` invoked `generate-pdf.mjs` with no arguments. The script printed its `Usage:` line and exited code 1. SPA toast still said "PDF generated" because the SSE `done` event fired regardless of exit code, but no file ever reached disk.
- **Documentation expansion** — 21-file `docs/test-scenarios/` directory in English, one file per page / endpoint, documenting goal + inputs + expected outputs + negative cases + the existing test that proves the contract.

## Findings closed

### 🔴 F-016 hardening — CV multipart upload silently corrupts `cv.md`

**Where:** `server/lib/routes/content.mjs` `app.post('/api/cv/import', express.raw(...))`.

**Reproduction.**
```
curl -X POST -F "file=@cv.md" http://127.0.0.1:4317/api/cv/import
# Response: { ok: true, markdown: "--------BoundaryXYZ\r\nContent-Disposition: form-data; ..."  }
```
The route blindly read `req.body` (already-parsed raw Buffer via `express.raw({ type: '*/*' })`) and passed it to `importDocumentToMarkdown`. For a `.md`-extension upload, the importer is a passthrough — so the multipart wire envelope (with `--boundary`, `Content-Disposition`, etc.) gets returned as if it were the file contents. The SPA then offers to Save, the user clicks, `cv.md` is overwritten with garbage.

The SPA's actual upload path uses `Content-Type: application/octet-stream` + `X-Filename: <name>` and POSTs the raw `ArrayBuffer` — so the SPA happy path was unaffected. Only external clients (curl `-F`, HTTP-client libraries that default to multipart for file uploads) triggered the corruption.

**Fix.**

1. Reject multipart explicitly at the route boundary: `if (Content-Type.startsWith('multipart/')) → 415 { error, hint }`.
2. Defense-in-depth sniff for `Content-Disposition: form-data` in the first 256 bytes of the body — catches misconfigured clients that send multipart bytes under a wrong Content-Type. Also → 415.
3. `cv.md` is never touched on either 415 path (route handler returns before any disk I/O).

**Verified.** Five new tests in `tests/cv-upload-multipart-reject.test.mjs`:

- SPA happy path returns 200 with clean markdown (no Content-Disposition leak).
- `multipart/form-data` Content-Type → 415 with hint.
- octet-stream BUT body bytes sniff as multipart → 415.
- Empty body → 400.
- Rejected request does NOT modify `cv.md` (read-back invariant).

### 🟠 PDF generation never produced a file (silent stream success)

**Where:** `server/lib/routes/runners.mjs` `app.get('/api/stream/pdf')`.

**Reproduction.**
```
curl -sN http://127.0.0.1:4317/api/stream/pdf
# event: start  data: {"script":"generate-pdf.mjs","args":[]}
# event: log    data: {"stream":"stderr","line":"Usage: node generate-pdf.mjs <input.html> <output.pdf> [--format=letter|a4]"}
# event: done   data: {"code":1}
```
The parent project's `generate-pdf.mjs` requires positional `<input.html> <output.pdf>` args. The route called it with `[]`. The script printed the usage line and exited code 1. The SPA's view consumed the `done` event without checking `code`, showed a green "PDF generated" toast, and the auto-download trigger silently no-op'd because no new file existed.

**Fix.**

1. Verify `cv.md` exists; if not, emit `event: error` + `event: done { code: 2 }` and close.
2. Render `cv.md` to HTML server-side via a small in-route helper `cvMarkdownToHtml(md, title)` — escapes everything first, then converts headings / lists / bold / italic / paragraphs, then wraps in a `<!doctype html>` shell with print-friendly CSS.
3. Write the HTML to `output/cv-input-<TIMESTAMP>.html`.
4. Spawn `generate-pdf.mjs <inputAbs> <outputAbs> --format=a4` (or `--format=letter` when `?format=letter`).

**Verified.** Three new tests in `tests/pdf-stream-args.test.mjs`:

- `event: start` carries `<input.html> <output.pdf> --format=a4` as **three positional args** with absolute paths; the input HTML exists on disk and contains the rendered `cv.md` content.
- `?format=letter` switches the third arg to `--format=letter`.
- Missing `cv.md` emits the expected error frame, not the misleading usage failure.

Manual end-to-end verification after the fix:

```
event: log  data: {"stream":"stdout","line":"📄 Input:  /…/output/cv-input-20260512T180749.html"}
event: log  data: {"stream":"stdout","line":"📊 Pages: 2"}
event: log  data: {"stream":"stdout","line":"📦 Size: 91.0 KB"}
event: done data: {"code":0}
```

`GET /api/output/pdfs` then returned the new file at the top. `GET /api/output/pdfs/cv-20260512T180749.pdf` served it with `Content-Disposition: attachment; filename="cv-20260512T180749.pdf"`. The SPA's auto-download trigger fires on the new file because `latestPdfName()` after `done` differs from the snapshot before `start`.
