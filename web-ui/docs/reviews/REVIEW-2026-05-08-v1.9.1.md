### 🟠 BF-1 (HIGH) — Tracker pipe-character injection in markdown table

**Where:** `server/lib/routes/tracker.mjs::POST /api/tracker` + `server/lib/parsers.mjs::parseMarkdownTable`.

**Problem:** Cells in `data/applications.md` are pipe-delimited GFM table rows. The route only escaped pipes in `notes` — `company`, `role`, and `reportSlug` were inserted raw. A name like `"Acme | Co"` would split the cell into two on the next read, corrupting the parse. Symmetric problem in the parser: it `.split('|')` ignored escapes, so even values written with the GFM-standard `\|` would still split.

**Fix:** Two-sided. Writer (tracker.mjs) now applies a single `cell()` sanitizer to every value: escape `|` → `\|`, collapse `\r\n` → space, trim. Reader (parsers.mjs) now uses a `splitUnescaped` helper that honors backslash-escapes, then unescapes `\|` → `|` per cell. End-to-end round-trip is now lossless even with wild names.

**Verified:** New Playwright test `tracker view renders empty + accepts API-seeded row` seeds `"Acme | Co"` and asserts the role survives the round-trip.

### 🟠 BF-2 (HIGH) — `POST /api/config` could throw uncaught on read-only filesystem

**Where:** `server/lib/routes/config.mjs::POST /api/config`.

**Problem:** `updateEnvFile` is a synchronous filesystem write. On a read-only mount, permission denied, or disk full, it throws — and the route had no `try/catch`, so Express would 500 with a stack trace. Worse: the secret-laden body was already in scope when the throw happened, so a sufficiently noisy error reporter could leak values.

**Fix:** Wrapped in `try/catch` that returns a clean 500 `{ error: 'failed to write parent .env', details: [e.message] }`.

### 🟠 BF-3 + BF-4 (MEDIUM) — Unbounded prompt size on Anthropic SDK calls

**Where:** `server/lib/routes/llm.mjs::/api/evaluate`, `/api/deep`, `/api/mode/:slug`.

**Problem:** `bundleProjectContext` truncates each individual file at 16 KB, but the assembled prompt can still grow large with many mode files + a 50 KB JD + the context block. Anthropic's largest current model (Sonnet 4.6 at 1M tokens) won't refuse it, but tokens cost money and a runaway prompt was a real footgun.

**Fix:** Soft cap at 200 KB (≈50K tokens) before the API call. On overflow: 413 `{ error: 'prompt too large', details: [<bytes> bytes; soft cap is <cap>] }`. Saves a multi-second roundtrip and prevents silent overspend.

### 🟢 BF-5+ (NIT — deferred)

The following were considered but not actioned in v1.9.1:

- **Activity-log rotation** — `data/activity.jsonl` grows forever. Documented as **P-13** in the v2.0.x roadmap.
- **Scan history rotation** — same shape; same rolling window, currently unbounded.
- **Backpressure on SSE** — if a client is on a slow link and the scanner spits out logs faster than the socket drains, Node buffers in memory. Not a real-world issue at current scale, but worth measuring.
- **CORS** — server explicitly does NOT set `Access-Control-Allow-Origin`. Same-origin is the only supported mode. Correct for single-tenant deployment; needs an explicit decision if anyone wants to call the API cross-origin.

These are NOT bugs; they are scope-defined limits to be revisited in v2.0.x.
