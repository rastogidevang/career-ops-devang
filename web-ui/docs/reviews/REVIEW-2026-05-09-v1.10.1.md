## Findings closed

### 🔴 F-003 (SECURITY, HIGH) — SSRF surface broader than docs claim

**Where:** [server/lib/security.mjs:37](../../server/lib/security.mjs#L37) `isValidJobUrl`; [server/lib/routes/pipeline.mjs:53](../../server/lib/routes/pipeline.mjs#L53) `/api/pipeline/preview`.

**Problem.** `isValidJobUrl` rejected `localhost`, `127.0.0.1`, `::1` only. RFC1918 ranges, AWS IMDS link-local 169.254.169.254, 0.0.0.0, CGNAT 100.64/10, IPv6 ULA / link-local all passed. Combined with the preview proxy that blindly trusted DNS, a malicious paste could probe the user's home LAN or — on hosts running `HOST=0.0.0.0` on AWS — exfiltrate IAM credentials via IMDS.

**Fix.** Extracted `isPrivateOrLoopbackHost(host)` that classifies all the above ranges (IPv4 + IPv6 + bracketed forms + `*.localhost`). `isValidJobUrl` now calls it. The preview proxy resolves the hostname per hop via `dns.lookup({verbatim:true})` and rejects when the resolved address itself is private — defeats DNS-rebind. DNS-failure case fails open (fetch will surface its own DNS error) so test stubs and DNS-less sandboxes still work.

**Verified.** `tests/url-validation.test.mjs` adds 5 new tests covering RFC1918, link-local, 0.0.0.0, 127/8, CGNAT, IPv6 ULA / link-local. `tests/critical-fixes.test.mjs` adds `isPrivateOrLoopbackHost` unit tests for IPv4 + IPv6 + bracketed forms.

### 🟠 F-005 — Activity log records 4xx-rejected attempts

**Where:** [server/lib/activity-log.mjs:104](../../server/lib/activity-log.mjs#L104) `activityMiddleware`.

**Problem.** `res.on('finish')` always logged, with `ok: false` on 4xx. Inflated the audit feed and let unauthenticated probes spam the log.

**Fix.** Middleware now early-returns when `res.statusCode >= 400`. The validate-write-record contract is implicit: only successful state changes get an event. `tests/activity-log.test.mjs` test 8 was the old contract; updated to assert the new behavior.

### 🟡 F-008 — `profile.save` not in activity feed

**Where:** [server/lib/activity-log.mjs:120](../../server/lib/activity-log.mjs#L120) `mapAction`.

**Problem.** The action map covered `pipeline`, `cv`, `jds`, etc., but not `/api/profile` (PUT) or `/api/config` (POST). Successful profile saves disappeared from the audit feed.

**Fix.** Added three entries: `/api/cv/import` → `cv.import`, `/api/profile` → `profile.save`, `/api/config` → `config.save`. Tested implicitly via the new `pickTarget` plumbing — no new test needed because the existing pipeline test already exercises the middleware.

### 🟡 F-009 — `/api/evaluate` ignores `mode:'manual'` opt-out

**Where:** [server/lib/routes/llm.mjs:53](../../server/lib/routes/llm.mjs#L53) `app.post('/api/evaluate')`.

**Problem.** When `ANTHROPIC_API_KEY` is set, every POST burned a real Anthropic call even when the SPA asked for `{ mode:'manual' }`. The `/api/deep` route already honored this flag.

**Fix.** `/api/evaluate` now mirrors `/api/deep`: if `mode === 'manual'`, return `{ mode:'manual', prompt, message }` immediately, skipping both the Anthropic and Gemini paths. Locale directive still applied so the manual prompt is in the user's language.

**Verified.** New test in `tests/critical-fixes.test.mjs`: `POST /api/evaluate { mode:'manual', lang:'ru' }` returns 200 with `mode:'manual'` and the prompt contains "Respond in Russian".

### 🟡 F-002 — Korean help body falls back to English

**Where:** [server/lib/routes/help.mjs:14](../../server/lib/routes/help.mjs#L14) `registerHelpRoutes`.

**Problem.** SPA's `I18n.getLang()` returns `'ko'` (bare BCP-47 code); the file on disk is `ko-KR.md`. The route tried `${lang}.md` then `en.md` — never tried `ko-KR.md` — so the EN bundle leaked into the Korean UI.

**Fix.** Resolver now walks four candidates: exact match, region-tagged alias from `FILE_ALIASES`, language-only base (`ko-KR` → `ko`), then `en.md`. The `FILE_ALIASES` map is the only place new bilingual aliases need to be added.

**Verified.** New test asserts `GET /api/help/ko` returns `lang: 'ko-KR'` with body > 1 KB; `/api/help/en` still returns `lang: 'en'`; unknown locale falls back to en.

### 🟡 F-017 — DELETE `/api/pipeline` shape inconsistency

**Where:** [server/lib/routes/pipeline.mjs:104](../../server/lib/routes/pipeline.mjs#L104).

**Problem.** Only honored `?url=`. Sent `body.url` was silently ignored. A miss returned 200 even when nothing was removed.

**Fix.** Reads URL from `req.query.url || req.body.url`. Returns 404 with `{ error, url }` when the URL isn't in the pipeline. Returns `{ ok:true, removed:1, url }` on success.

**Verified.** Three new tests in `tests/critical-fixes.test.mjs` cover the body path, the 404 path, and the 400-no-url path.

### 🟡 F-012 — LLM prompts ignore SPA locale

**Where:** [server/lib/prompts.mjs](../../server/lib/prompts.mjs); [server/lib/routes/llm.mjs](../../server/lib/routes/llm.mjs); [public/js/api.js](../../public/js/api.js).

**Problem.** Mode templates and the evaluation/deep prompts were always English-only. A Russian user typing into `#/evaluate` got an EN-language response back from Anthropic.

**Fix.** Three pieces:
1. `prompts.mjs` exports `resolveLocale(req)` (precedence: `body.lang` → `body.locale` → `Accept-Language` header → `'en'`) and `buildLocaleDirective(lang)` (returns one-line "Respond in X" or empty for `en`).
2. `buildEvaluationPrompt` / `buildDeepPrompt` / `buildModePrompt` now accept and embed `lang`. `buildModePrompt` also strips `lang` and `locale` from the rendered JSON context block.
3. `public/js/api.js` `call()` attaches `Accept-Language: <currentLocale>` to every request and merges `lang: <currentLocale>` into JSON bodies (when not already present).

**Verified.** Five new tests in `tests/critical-fixes.test.mjs`: `resolveLocale` precedence, `buildLocaleDirective` outputs, evaluation / deep / mode prompts each embed the directive when non-en.
