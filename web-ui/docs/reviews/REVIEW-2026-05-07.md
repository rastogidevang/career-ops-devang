## 🟠 HIGH

### A1 — Anthropic live mode produces hollow output (no file context inlined)

**Where:** `server/index.mjs::buildDeepPrompt`, `buildModePrompt`; both fed into `runAnthropic(prompt)` from `server/lib/anthropic.mjs`.

**Problem:** Both prompt builders include a literal `"Read these files first: cv.md, config/profile.yml, modes/_shared.md, modes/<slug>.md"` instruction. That works inside Claude Code (filesystem-aware) and inside `gemini-eval.mjs` (a Node script with read access in the parent project). It does **not** work in the Anthropic SDK call path: `runAnthropic` POSTs to `/v1/messages` with a single `{ role: 'user', content: prompt }`. The model has zero filesystem access. So when the SPA's "Run live" button fires `/api/deep { run: true }` or `/api/mode/<slug> { run: true }` and the user has only `ANTHROPIC_API_KEY` set, the model fabricates from the prompt template alone — no CV match, no profile, no mode-specific structure beyond the template inlined here.

**Fix:** Wrap a small `bundleProjectContext({ slug })` helper that reads `PATHS.cv`, `PATHS.profile`, `modes/_shared.md`, and `modes/<slug>.md` (where applicable), and prepends a clearly delimited `<context>...</context>` block before the prompt body. Apply in both `/api/deep` and `/api/mode/:slug` Anthropic branches. Cap each file at ~16 KB for safety.

**Impact:** Live deep-research and live mode prompts produce real, grounded answers. Test today vs after fix on the same query — the difference is qualitative, not quantitative.

### A2 — `runNodeScript` and `streamNodeScript` send SIGTERM only, no SIGKILL escalation

**Where:** `server/lib/runner.mjs:23-27` (buffered) and `:76-91` (streaming).

**Problem:** Both timer / cleanup paths send `SIGTERM` and rely on the child to honor it. A child stuck in a syscall (DNS, blocked socket read, `await fetch` to a stalling host) ignores `SIGTERM` until it returns. Result: the runner promise never resolves; for SSE, the connection slot stays held until Node's GC eventually reaps. Under load this leaks file descriptors.

**Fix:** Add a watchdog. After `SIGTERM`, set a `setTimeout` (e.g. 5 s) that, if `child.killed === false` and `close` hasn't fired, sends `SIGKILL` and resolves the promise with `{ code: -1, killed: true }`. Same shape for the streaming variant.

---

### A3 — `streamNodeScript` has no max-runtime cap

**Where:** `server/lib/runner.mjs:44-91`.

**Problem:** Streaming endpoints (`/api/stream/scan`, `/api/stream/liveness`, `/api/stream/pdf`) inherit no upper bound — they live until natural exit, child error, or client disconnect. A scan script with a redirect loop or hung HTTP can keep the SSE channel open indefinitely.

**Fix:** Pass `opts.maxRuntimeMs` (default 30 minutes for SSE, 60 s for buffered already exists). On expiry: emit `event: error { message: 'maximum runtime exceeded' }`, kill child via the A2 watchdog, end the response.

## 🟡 MEDIUM

### B1 — `/api/pipeline/preview` follows redirects without per-hop validation

**Where:** `server/index.mjs:404-438`.

**Problem:** `isValidJobUrl()` validates the input URL (rejects loopback, `file://`, etc.). The fetch then uses `redirect: 'follow'`, which follows up to 20 redirects by default. A hostile board could redirect to `http://localhost/` or a private IP — `isValidJobUrl` was bypassed for those hops. Mitigated only by network egress not reaching most internals on a default loopback bind, but on `HOST=0.0.0.0` deployments the gap widens.

**Fix:** Switch to `redirect: 'manual'` and walk redirects in code, applying `isValidJobUrl` to each `Location` header. Cap at 3 hops.

### B2 — Mixed Anthropic key checks (`hasAnthropicKey()` vs raw `process.env`)

**Where:** `server/index.mjs:791, 819, 885, 902` mix `hasAnthropicKey()` and direct env lookups for `GEMINI_API_KEY`. Minor drift; standardize on the helper for both providers.

**Fix:** Add `hasGeminiKey()` to `lib/anthropic.mjs` (rename file or move helper) or to a new `lib/llm-keys.mjs`. Replace direct `process.env.GEMINI_API_KEY` reads in `index.mjs`. Keeps test mocks consistent.

### B3 — In-process scanners ignore client disconnect for in-flight fetches

**Where:** `server/lib/ru-scanner.mjs::runRuScan`, `server/lib/en-scanner.mjs::runEnScan`.

**Problem:** SSE handlers for `/api/stream/scan-{ru,en}` capture `aborted = true` on `res.close`, then drop event emissions. But the scanner functions don't accept an `AbortSignal` — their inner `fetch` calls keep running. After client disconnect, the work continues until the queries iterator finishes.

**Fix:** Thread `signal` through `runRuScan` / `runEnScan` → `searchHH` / `searchHabr` / `fetchCompanyJobs(...)` and pass to `fetch`. Honor in the per-query loop.

---

### B4 — Anthropic adapter has no `Authorization`-style logging guard

**Where:** `server/lib/anthropic.mjs:29-42`.

**Problem:** `runAnthropic` constructs a request with `x-api-key` set from `process.env.ANTHROPIC_API_KEY`. If a future change adds `console.log(JSON.stringify(opts))` inside this function, the key leaks to stdout. The activity log middleware doesn't see this call (it's outbound, not inbound). Defensive: stamp a `// SECURITY: do not log opts or req` comment, and add a unit test that asserts `console.log` is not called during a successful run.

**Fix:** Add the comment + a test using a `--test --node-options="--unhandled-rejections=strict"` style spy on `console.log`. Cheap.
