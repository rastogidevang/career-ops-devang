## Notable design decisions

### G-014 — Short-circuit at the top of the handler, not inside the `evaluate` step

Pre-v1.25 the route's manual-mode path was buried inside step 3 (evaluate). When the user passed `mode: 'manual'`, the handler still ran step 1 (validate URL — fine), step 2 (fetch JD via `safeGet` — burns ~1 s and DNS lookup), then hit step 3 and decided manual mode was incompatible (the `evalMode === 'manual'` branch threw "no LLM key set; manual mode incompatible with auto-pipeline"). Result: request hung for the fetch round-trip and returned a failure instead of a usable prompt.

v1.25 moves the short-circuit before step 2. As soon as URL validation passes:

- Emit `step` events for stages 1-5 with status `done` / `skipped (manual mode)`.
- Build the canonical evaluate prompt via `buildEvaluationPrompt('', lang)`. Empty JD string is intentional — the user will paste it themselves when running the prompt in Claude Code / Anthropic / Gemini.
- Send `done` event with `{ mode: 'manual', prompt, message }`.
- End the response.

No fetch, no DNS lookup beyond validation, no LLM call. Response returns in < 200 ms on the test fixture.

### G-014 — Accept both `mode` and `evalMode` for back-compat

The FIX-PROMPT specified `mode: 'manual'` (mirroring `/api/evaluate`'s contract from v1.10.2). The pre-v1.25 code already accepted an `evalMode` override (line 161). v1.25 honours BOTH spellings. Any in-flight client still passing `evalMode: 'manual'` keeps working. Test `tests/auto-pipeline-manual-mode.test.mjs` has a dedicated case for the legacy spelling.

### G-012 — Parallel translation agents + CI gate

CHANGELOG locale drift had been accumulating since v1.23 (the v1.23 cycle landed entries in RU only; v1.24 + v1.24.1 entries never propagated). The fix uses the parallel-agents pattern that's worked for prior locale-content refreshes (README quality refresh in v1.22, help-bundle refresh in v1.24): 7 agents in flight, each receiving the 3-4 missing entries from the EN canonical and translating into its target language.

To prevent recurrence: new `scripts/check-changelog-parity.mjs` CI gate reads the newest `## [X.Y.Z]` heading from each CHANGELOG file and fails the build if any locale lags behind EN. Wired into `npm run test:ci`. Next release with locale drift fails at CI time, not at "user notices the EN-only entry six months later" time.

### G-003 — Stale finding verification

The FIX-PROMPT flagged `README.cn.md` → `README.zh-CN.md` as needed. `find` confirmed no `README.cn.md` exists anywhere in the worktree, the repo already has `README.zh-CN.md`, and every reference in `README.md` + `docs/*.md` points at the `zh-CN` name. The G-003 finding was stale — likely from a pre-v1.20 snapshot before the rename happened. Closed with a note, no code change.

### G-005 — Honest deferral

Realigning the report block schema from our A-G (with separate Risks + Verdict + Legitimacy blocks) to the canonical A-F (Strategy / Personalization / STAR stories) requires rewriting `modes/oferta.md` in the parent `santifer/career-ops` project. v1.25 ships the web-ui side ready: `reports.js` since v1.13 has accepted arbitrary block letters, `prompts.mjs::SCAFFOLD_STRINGS` is locale-aware, and the help-bundle §9 sections describe Evaluate blocks in a schema-agnostic way. When the parent commit lands, we'll need a coordinated tag bump (web-ui + career-ops together) and a CHANGELOG entry that calls out the migration. Tracking as v1.26 candidate.

### `✨ ✨` cosmetic — fix the data, not the view

The dashboard renderer in `public/js/views/dashboard.js:58` prepends `'✨ '` to the translation. Every locale value in `dash.autoPipeline` ALSO began with `'✨ '`. Two sources of truth, both contributing the same glyph, no obvious owner. v1.25 drops the leading `'✨ '` from every locale string — the view's prefix is the single source. Audit pass found no other double-glyph patterns in the i18n bundle (checked `📁`, `▶`, `🌐`, `📄`, `💾`).
