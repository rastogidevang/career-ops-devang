## Notable design decisions

### `htmlFor` alias in `UI.el()` instead of bespoke `setLabelFor` helper

`for` is JS-reserved as a property name on Element, so setting it via
the el-builder requires either a string-literal property (which Object
spread handles fine) or a dedicated attribute path. We chose to mirror
React's `htmlFor` alias because every view that needs label-to-input
association reads the same way it would in JSX. The translation lives
in one place in [`public/js/api.js`](../../public/js/api.js): if
`htmlFor` is set, `el()` calls `el.setAttribute('for', v)` and skips
the property assignment. Existing call sites that used the raw `for`
attribute via `setAttribute` still work; nothing was deprecated.

### Chip touch-target — `min-height: 28px` + `gap: 8px` (not `min-height: 44px`)

`.chip` clusters on Scan + Tracker render 4–12 chips in a tight row.
Bumping each to 44 × 44 px would visually destroy the chip-row
aesthetic. WCAG 2.5.8 (the AA-AAA bridge that supersedes 2.5.5 for
target spacing) allows targets < 24 × 24 px IF they have ≥ 24 px of
clearance to the nearest other target. We chose the spaced-target
exception: 28 px tall (with 12 px horizontal padding so the click
zone is wider than the visible chip), and `.chip-row { gap: 8px }` on
the wrapper. Combined, every chip's center is ≥ 36 px from its
neighbor's center — well above the 24 px floor.

### Mode-page form fields — `htmlFor` only, no inline hint copy yet

The generic mode-page factory currently exposes `<label>` + `<input>`
pairs without inline-hint paragraphs. v1.20.0 wires up the
label-to-input association (`htmlFor` → stable `id`) — the minimum to
satisfy WCAG 3.3.2 — but stops short of writing per-field hint copy
across 7 modes × ~3 fields each × 8 locales (≈ 168 new i18n keys).
That polish item is documented in the v1.20 "Out of scope" table for
v1.21+. The README walkthroughs already document each mode's field
intent in every locale, so users have the explanation; the SPA just
doesn't surface it inline yet.

### Non-EN README parity — full rewrite, not patch

The 7 non-EN READMEs were 306–316 lines vs. EN's 585. They covered
the headlines but skipped the marketing-heavy walkthroughs (Why?,
Quick start 3-step, 🌍 Getting Started 5-step) and most of the API
reference. v1.20 dispatched 7 parallel translation subagents — one
per locale — each given the EN master and the current locale file as
tone reference. Each subagent produced a full-file rewrite that
mirrors EN structure end-to-end (About → install → Why? → Quick
start → Requirements → What you get → Scan → Architecture → API
reference → Tests → Configuration → Security → Limitations →
Contributing → Getting Started → License). Code blocks (commands,
YAML, env vars), URLs, and shields.io badge URLs are preserved
verbatim.

### `/api/scan-ru/config` — removed as scheduled, no further alias

v1.19.0 introduced the canonical `/api/scan/regional/config` path
and kept `/api/scan-ru/config` as a one-release alias for back-compat.
v1.20.0 removes the alias outright (no RFC 8594 Sunset header phase
this time — the v1.19 CHANGELOG and the inline header comment both
flagged the sunset version explicitly, and the canonical path has been
documented in every locale README since v1.19).
